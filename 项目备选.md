# 大模型微调项目备选

## 选题标准

当前阶段不再以“跑通框架”为目标，而是选择一个值得深入钻研的真实项目题目。优先考虑：

- 有公开数据集或真实 GitHub 项目可以参考。
- 能自然拆成 SFT + Reward Model / Preference Learning / RL 阶段。
- 有明确评价指标，避免泛聊天微调之后不知道好在哪里。
- 问题本身有研究价值或工程落地价值。

推荐避免一开始做纯泛聊天助手微调，因为目标太散，SFT 后面的 reward/RL 阶段容易变成形式主义。

## 备选方向总览

| 方向 | 真实项目/数据 | 适合钻研的问题 | 推荐程度 |
|---|---|---|---|
| 数学推理 RLVR | Open-R1、AIMO/Kaggle 数学题、GSM8K、MATH | SFT 和 RL 分别提升什么；可验证 reward 如何设计 | 高 |
| RAG 回答对齐 | RAG-RewardBench、DPA-RAG、RAGferee | 如何减少幻觉；如何让回答忠于检索材料 | 高 |
| 安全对齐 | PKU-SafeRLHF / Safe-RLHF | helpfulness 和 safety 的冲突；安全偏好建模 | 中高 |
| 摘要 RLHF | OpenAI summarize-from-feedback | 经典 SFT -> RM -> PPO 链路复现 | 中高 |
| 真实聊天偏好建模 | LMSYS Chatbot Arena / Kaggle 偏好数据 | reward model 偏好学习；长度偏好、风格偏好、模型身份偏差 | 中 |
| 工具调用 / Agent RL | ToolBench、Tool-Star、Agent Lightning | SFT 学工具调用轨迹，RL 优化任务成功率 | 中 |
| 金融领域微调 | FinGPT / FinLoRA | 金融问答、情绪分析、报告理解等领域微调 | 中 |

## 方向一：数学推理 RLVR

### 项目题目

基于 Qwen 小模型的数学推理 SFT + GRPO/RLVR 训练：比较纯 SFT、SFT + RL、不同 reward 设计对 GSM8K / MATH / AIMO 类任务的影响。

### 参考项目与数据

- Open-R1: https://github.com/huggingface/open-r1
- Open-R1 博客: https://huggingface.co/blog/open-r1
- DeepSeek-R1 技术方向参考: https://github.com/deepseek-ai/DeepSeek-R1
- GSM8K: https://huggingface.co/datasets/openai/gsm8k
- MATH: https://huggingface.co/datasets/hendrycks/competition_math
- AIMO / Kaggle 数学竞赛方向: https://www.kaggle.com/competitions/ai-mathematical-olympiad-prize

### 为什么值得做

这个方向的 reward 相对干净：答案对不对、格式是否正确、最终答案是否可解析。它不像聊天偏好那样主观，比较适合研究 SFT 和 RL 的真实差异。

可以重点研究：

- 纯 SFT 能否提升解题格式和基础能力。
- RLVR 是否能进一步提升正确率。
- 只奖励 final answer 和同时奖励格式/推理过程，效果有什么区别。
- 小模型是否会出现 reward hacking。
- 数据难度分布对 RL 效果的影响。

### 可做实验

- Base Model vs SFT Model vs SFT + RL Model。
- 不同 reward：答案正确奖励、格式奖励、长度惩罚、步骤可解析奖励。
- 不同训练数据规模：1k、5k、10k。
- 不同模型大小：0.5B、1.5B、3B。

### 适合产出

- 一篇完整实验报告。
- 一套可复现实验脚本。
- 一个数学推理小模型 checkpoint。
- 一组关于 SFT/RL 差异的定量分析。

## 方向二：RAG 回答对齐

### 项目题目

面向 RAG 的奖励模型训练：让小模型学会基于检索材料回答、拒答无依据问题、减少幻觉。

### 参考项目与数据

- RAG-RewardBench: https://github.com/jinzhuoran/RAG-RewardBench
- RAG-RewardBench 数据: https://huggingface.co/datasets/jinzhuoran/RAG-RewardBench
- DPA-RAG: https://github.com/dongguanting/DPA-RAG
- RAGferee 论文方向: https://arxiv.org/abs/2509.26011

### 为什么值得做

这个方向工程价值强，适合企业知识库、文档问答、客服、报告问答等场景。SFT 可以学习回答格式，Reward Model 可以学习“答案是否忠于上下文”，RL 阶段可以继续优化答案质量。

可以重点研究：

- 回答是否被检索材料支持。
- 无依据时是否拒答。
- 模型是否会编造引用。
- reward model 能否识别忠实回答和幻觉回答。
- 偏好数据如何构造：人工标注、规则合成、强模型打分。

### 可做实验

- 构建一个小型 RAG 数据集：query、context、chosen answer、rejected answer。
- 训练 Reward Model 判断回答是否忠于 context。
- SFT 后对比 RL 前后幻觉率。
- 引入引用检查、答案覆盖率、拒答率等指标。

### 适合产出

- 一个 RAG 对齐训练项目。
- 一个 reward model，用于给 RAG 答案打分。
- 一套幻觉检测和忠实性评估脚本。
- 可落地到知识库问答系统。

## 方向三：安全对齐 / Safe RLHF

### 项目题目

基于 PKU-SafeRLHF 的安全对齐实验：比较 SFT、Reward Model、Cost Model、PPO-Lagrange 对 helpfulness/safety 的影响。

### 参考项目与数据

- Safe-RLHF: https://github.com/PKU-Alignment/safe-rlhf
- PKU-SafeRLHF 数据: https://huggingface.co/datasets/PKU-Alignment/PKU-SafeRLHF
- 论文: https://arxiv.org/abs/2406.15513

### 为什么值得做

这个方向更偏 alignment 研究。它不是单纯提升任务能力，而是研究模型如何在“有用”和“安全”之间权衡。

可以重点研究：

- 安全训练是否会损伤 helpfulness。
- 模型是否出现过度拒答。
- Reward Model 和 Cost Model 的分工。
- 不同安全策略对回答风格的影响。

### 可做实验

- SFT only vs SFT + RM vs SFT + RM + Cost Model。
- 比较普通 PPO 和 PPO-Lagrange。
- 统计拒答率、有害回答率、有用性评分。

### 适合产出

- 一份安全对齐实验报告。
- 一个小规模安全 reward/cost model。
- 对 helpfulness-safety trade-off 的分析。

## 方向四：摘要 RLHF

### 项目题目

复现 OpenAI summarize-from-feedback：基于人类偏好数据训练摘要模型，完成 SFT -> Reward Model -> PPO 链路。

### 参考项目与数据

- OpenAI summarize-from-feedback: https://github.com/openai/summarize-from-feedback
- TRL StackLLaMA / RLHF 参考: https://huggingface.co/blog/stackllama

### 为什么值得做

这是非常经典的 RLHF 任务。摘要有明确输入文本和输出摘要，偏好数据也更容易理解，比开放聊天更容易分析。

可以重点研究：

- 人类偏好是否偏向更长、更具体或更流畅的摘要。
- Reward Model 是否学到长度偏置。
- PPO 是否真的提升人工偏好指标。
- 自动指标 ROUGE 和 reward 分数是否一致。

### 可做实验

- SFT 摘要模型。
- Reward Model 训练 chosen/rejected 摘要。
- PPO 优化摘要质量。
- 比较 ROUGE、reward score、人工偏好。

### 适合产出

- 一个经典 RLHF 复现实验。
- 一份关于 reward model 偏差的分析。
- 一套摘要质量评估脚本。

## 方向五：真实聊天偏好建模

### 项目题目

基于 LMSYS Chatbot Arena 偏好数据训练 Reward Model，研究人类偏好中的长度偏好、风格偏好和模型身份偏差。

### 参考项目与数据

- LMSYS Chatbot Arena Kaggle 竞赛: https://www.kaggle.com/competitions/lmsys-chatbot-arena
- LMSYS 竞赛介绍: https://www.lmsys.org/blog/2024-05-02-kaggle-competition/

### 为什么值得做

这是比较真实的人类偏好数据，适合训练 reward model 或偏好预测模型。缺点是它更适合偏好建模，不一定天然包含完整 SFT + RL 链路。

可以重点研究：

- 人类是否偏好更长回答。
- 模型身份是否影响偏好。
- 不同任务类型下偏好是否不同。
- Reward Model 是否能泛化到新模型输出。

### 可做实验

- 用 pairwise 数据训练 Reward Model。
- 分析长度、语气、结构对胜率的影响。
- 将 Reward Model 用于 rerank 多个候选回答。

### 适合产出

- 一个偏好预测模型。
- 一份聊天偏好分析报告。
- 一个回答 reranker 原型。

## 方向六：工具调用 / Agent RL

### 项目题目

基于工具调用轨迹的 SFT + RL 训练：让模型学会选择工具、调用 API，并用任务成功率作为奖励信号。

### 参考项目与数据

- ToolBench: https://github.com/OpenBMB/ToolBench
- Agent Lightning: https://github.com/microsoft/agent-lightning

### 为什么值得做

这是比较前沿的方向，适合研究 Agent 和工具调用。SFT 可以学习工具调用格式和轨迹，RL 可以优化最终任务成功率。

难点是环境搭建和奖励定义会更复杂，不建议作为第一个深入项目，适合作为第二阶段。

可以重点研究：

- 工具选择是否正确。
- 参数生成是否可靠。
- 多步工具链是否能完成任务。
- 最终任务成功率是否能作为 reward。

### 可做实验

- SFT 学习工具调用轨迹。
- 构建可自动验证的工具任务。
- 使用成功/失败作为 reward。
- 比较 SFT 和 RL 后任务完成率。

### 适合产出

- 一个工具调用微调项目。
- 一个 agent 任务评测环境。
- 一份关于工具调用错误类型的分析。

## 方向七：金融领域微调

### 项目题目

基于 FinGPT / FinLoRA 的金融领域指令微调与偏好优化：面向金融问答、情绪分析、公告理解或报告摘要。

### 参考项目与数据

- FinGPT: https://github.com/AI4Finance-Foundation/FinGPT
- FinRL: https://github.com/AI4Finance-Foundation/FinRL

### 为什么值得做

金融方向有真实业务感，适合做领域模型。但需要注意数据泄漏、时序切分、事实性和投资建议风险。它更适合领域 SFT，不一定天然适合完整 RLHF，除非自己构造偏好数据。

可以重点研究：

- 金融新闻情绪分类。
- 财报/公告问答。
- 研报摘要。
- 金融问答的拒答边界。

### 可做实验

- 领域 SFT：金融 QA、摘要、情绪分析。
- 构造 chosen/rejected 偏好数据：事实正确 vs 编造、谨慎表达 vs 过度建议。
- 训练 Reward Model 约束事实性和风险表达。

### 适合产出

- 一个金融领域小模型。
- 一个金融问答/摘要 demo。
- 一份金融事实性和风险控制分析。

## 竞赛型新增备选

这一部分更偏“真实赛题反推项目”。不是所有竞赛都天然适合完整 SFT + RLHF，但可以改造成 SFT、Reward Model、DPO/RLVR 或 RAG reward 项目。

### Kaggle：AI Mathematical Olympiad / AIMO

项目题目：

> 基于 AIMO / 数学竞赛题的推理模型训练：SFT 学习长推理轨迹，DPO/RLVR 优化答案正确率和输出长度。

参考来源：

- Kaggle AIMO: https://www.kaggle.com/competitions/ai-mathematical-olympiad-prize
- AIMO2 二等奖方案: https://github.com/imagination-research/aimo2
- OpenR1-Math 数据参考: https://huggingface.co/datasets/open-r1/OpenR1-Math-220k

适合原因：

- 真实竞赛题，评价指标清楚。
- 可验证答案天然适合 RLVR。
- 参考方案里包含 SFT、DPO、推理投票、长度控制等内容。

可以研究：

- SFT 是否主要提升格式和推理展开。
- DPO 是否能减少冗长输出。
- RLVR 是否能稳定提升最终答案正确率。
- 多采样投票和训练本身谁更有效。

推荐程度：很高。

### Kaggle：Eedi - Mining Misconceptions in Mathematics

项目题目：

> 面向数学教育的误解识别模型：用 SFT/Reranker 学生错因诊断，并用偏好数据优化误解候选排序。

参考来源：

- Kaggle Eedi: https://www.kaggle.com/competitions/eedi-mining-misconceptions-in-mathematics
- Silver Medal 方案: https://github.com/DaoyuanLi2816/kaggle-eedi-mining-misconceptions-in-mathematics-silver-medal
- Eedi 官方介绍: https://eedi.com/us/blog/from-wrong-answers-to-real-insights-how-we-used-a-kaggle-challenge-to-map-student-misconceptions

适合原因：

- 真实教育场景，不是玩具任务。
- 数据包含题目、错误选项、误解标签，适合做检索 + 重排。
- 可以把 reranker 训练改造成 Reward Model：对正确误解排序更高，对错误误解排序更低。

可以研究：

- Qwen/Embedding 检索候选 + LLM rerank 的组合。
- 数学错误原因解释能力是否能通过 SFT 提升。
- pairwise ranking loss / DPO 是否能改善误解排序。
- 模型是否能泛化到未见过的 misconception。

推荐程度：高。它比纯数学解题更偏教育 AI，题目很有辨识度。

### Kaggle：LMSYS Chatbot Arena Human Preference Predictions

项目题目：

> 基于 Chatbot Arena 的真实人类偏好 Reward Model：研究回答长度、风格、任务类型对人类偏好的影响。

参考来源：

- Kaggle LMSYS: https://www.kaggle.com/competitions/lmsys-chatbot-arena
- LMSYS 官方介绍: https://www.lmsys.org/blog/2024-05-02-kaggle-competition/
- 第四名方案: https://github.com/DaoyuanLi2816/Kaggle-4th-Place-Solution-LMSYS-Chatbot-Arena-Human-Preference-Predictions

适合原因：

- 真实人类偏好数据，天然适合 Reward Model。
- 可以接到 DPO/RLHF：用 RM 给多个回答排序，再做偏好优化。
- 很适合写偏好分析报告。

可以研究：

- 人类是否偏好更长回答。
- 不同模型身份是否影响胜率。
- reward model 是否学到风格偏置。
- 用 RM rerank 多候选回答是否能提升偏好分。

推荐程度：高，但更偏 Reward Model / 偏好建模，不是完整任务型 SFT。

### Kaggle：LLM Prompt Recovery

项目题目：

> 提示词反推模型：给定原文和改写文本，微调小模型恢复改写 prompt。

参考来源：

- Kaggle LLM Prompt Recovery: https://www.kaggle.com/competitions/llm-prompt-recovery
- 完整 notebook 教程: https://alex-yang-upenn.github.io/kaggle-prompt-recovery/intro.html
- GitHub 参考: https://github.com/jjia1/llmpromptrecovery_kaggle

适合原因：

- 是一个生成式任务，适合直接做 SFT。
- 参考教程里有生成训练数据、LoRA/QLoRA 微调、评估全流程。
- 适合研究“模型是否能学会风格转换背后的隐含指令”。

可以研究：

- 合成数据质量对 prompt recovery 的影响。
- SFT 模型和纯 prompt 工程的差距。
- 能否用 Reward Model 评价恢复 prompt 和真实 prompt 的相似性。

推荐程度：中高。题目有趣，但 reward/RL 部分需要自己扩展。

### Kaggle：LLM - Detect AI Generated Text

项目题目：

> AI 生成文本检测与反检测数据生成：训练检测器，同时用微调生成模型生成更像学生写作的样本。

参考来源：

- Kaggle 竞赛: https://www.kaggle.com/competitions/llm-detect-ai-generated-text
- 第一名方案: https://github.com/rbiswasfc/llm-detect-ai

适合原因：

- 可做安全/检测方向。
- 第一名方案里包含用多种 LLM 在 PERSUADE 语料上继续训练，生成学生风格作文。
- 可以做成“生成器 + 判别器”的对抗式项目。

可以研究：

- 微调生成模型后，文本是否更难被检测。
- 检测器对不同模型来源文本的泛化。
- 用检测器作为 reward，是否会让生成模型逃避检测。

推荐程度：中。更偏检测和安全，不是标准 SFT + RLHF，但能做出有意思的研究。

### Kaggle：Jigsaw 社区规则/毒性检测

项目题目：

> 面向社区治理的安全 Reward Model：判断评论是否违反社区规则，并研究不同社区规范下的安全对齐。

参考来源：

- Jigsaw Agile Community Rules Classification: https://www.kaggle.com/competitions/jigsaw-agile-community-rules
- CompeteHub 介绍: https://www.competehub.dev/en/competitions/kagglejigsaw-agile-community-rules
- Jigsaw 系列开源方案参考: https://github.com/vykhand/kaggle-jigsaw

适合原因：

- 真实内容审核场景。
- 可以训练 safety classifier / cost model。
- 能和 SafeRLHF 类方向接上。

可以研究：

- 不同社区规则下，什么算违规。
- 安全模型是否会误伤正常表达。
- 用 cost model 约束生成模型，是否降低违规率。

推荐程度：中。适合做安全方向，不适合纯生成能力提升。

### 天池：通义千问 AI 挑战赛 - Code Qwen 能力算法赛道

项目题目：

> 面向代码能力的 Qwen 微调：通过高质量代码数据提升算法解题、漏洞修复或代码生成能力。

参考来源：

- 通义千问 AI 挑战赛: https://tianchi.aliyun.com/specials/promotion/Qwen
- Code Qwen 赛道: https://tianchi.aliyun.com/competition/entrance/532169/introduction
- Qwen2.5-Coder 技术方向: https://github.com/QwenLM/Qwen2.5-Coder

适合原因：

- 官方赛道明确写的是“对通义千问大模型的微调训练”。
- 代码任务容易构造可验证 reward：单测是否通过、答案是否 AC、补丁是否正确。
- 可以自然扩展成 SFT + RLVR：SFT 学代码解法，RL 用测试结果奖励。

可以研究：

- 用 Codeforces/LeetCode/MBPP/HumanEval 构造 SFT 数据。
- 用单元测试作为 reward。
- 比较普通 SFT、DPO、RLVR 对代码通过率的影响。

推荐程度：很高。如果你想做“竞赛 + 真实能力 + 可验证 reward”，这是国内平台里非常合适的方向。

### 天池 / SMP：ChatGLM 金融大模型挑战赛 / FinGLM

项目题目：

> 基于上市公司年报的金融问答模型：SFT 学金融问答，RAG 检索年报证据，Reward Model 约束事实性和数值正确性。

参考来源：

- FinGLM: https://github.com/MetaGLM/FinGLM
- SMP 2023 ChatGLM 金融大模型挑战赛: https://tianchi.aliyun.com/specials/promotion/SMP2023ChatGLMChallenge

适合原因：

- 有真实竞赛背景、真实年报数据、开源方案和评测数据。
- FinGLM 项目包含数据准备、PDF/TXT/HTML 年报、数据库、模型微调、问答流程。
- 适合做 RAG + SFT + Reward Model 的综合项目。

可以研究：

- 年报 PDF 解析质量对问答效果的影响。
- SFT 是否能提升金融问答格式和领域术语。
- Reward Model 是否能识别数值错误、无依据回答。
- RAG 和微调分别解决什么问题。

推荐程度：很高。这个项目最像“可落地的中文真实项目”。

### 天池：全球智能汽车 AI 挑战赛 - 大模型检索问答

项目题目：

> 面向智能汽车知识库的 RAG 问答模型：基于汽车领域文档构建检索问答，并训练答案质量 Reward Model。

参考来源：

- Tianchi-LLM-QA baseline: https://github.com/dawoshi/Tianchi-LLM-QA
- 赛事介绍参考: https://segmentfault.com/a/1190000046079924

适合原因：

- 真实行业 RAG 场景。
- 有公开 baseline 可参考。
- 适合研究“领域知识问答中，微调和检索增强的边界”。

可以研究：

- 汽车文档切分、召回、重排。
- SFT 让模型学会基于资料回答。
- RM 判断答案是否被资料支持。
- RL/DPO 优化忠实性和引用格式。

推荐程度：中高。更工程化，研究味略弱于 AIMO/Eedi，但落地感强。

### 天池：中医 AI / 杏林机甲 LLM 微调赛

项目题目：

> 中医问答领域微调：基于中医问答和知识数据训练垂直模型，并研究专业性、谨慎性和幻觉控制。

参考来源：

- 天池中医 AI 专题学习赛: https://tianchi.aliyun.com/competition/entrance/532204
- 扁仓中医大模型参考: https://github.com/QLU-NLP/BianCang-TCM-LLM
- TCMLLM 参考: https://github.com/2020MEAI/TCMLLM

适合原因：

- 中文垂直领域，适合做领域 SFT。
- 医疗相关任务天然需要安全边界和拒答策略。
- 可以构造偏好数据：专业、谨慎、有依据的回答优于武断、编造、过度诊断的回答。

可以研究：

- 小模型是否能学会中医术语和问答风格。
- RAG 是否比直接微调更适合专业知识。
- Reward Model 是否能约束医疗幻觉和过度建议。

推荐程度：中高。题目有特色，但要注意医疗安全边界，项目表述最好定位为“学习研究”，不要包装成诊疗工具。

## 竞赛型方向优先级

如果只从“适合你现在做完整训练研究”的角度排，我建议：

1. AIMO / 数学推理 RLVR：最适合 SFT + DPO/RLVR，指标清楚。
2. Code Qwen / 代码能力微调：适合 SFT + 单测 reward，工程和研究都强。
3. FinGLM / 金融年报问答：最适合中文真实项目，适合 RAG + SFT + RM。
4. Eedi / 数学教育误解识别：题目独特，适合检索 + rerank + 偏好优化。
5. LMSYS / Chatbot Arena：最适合 Reward Model 和偏好分析。
6. 汽车 RAG 问答：工程落地好，适合做知识库方向。
7. 中医微调：中文垂直领域有特色，但安全表述要谨慎。
8. Prompt Recovery：适合 SFT 学习，RL/RM 需要自己设计。
9. Jigsaw / AI 文本检测：适合安全和检测方向，不是最核心的生成式微调。

## 推荐选择

如果目标是前沿和研究价值，优先选：

1. 数学推理 RLVR / Open-R1 路线。
2. RAG 回答对齐 / RAG Reward 路线。
3. 安全对齐 / PKU-SafeRLHF 路线。

如果目标是工程落地，优先选：

1. RAG 回答对齐。
2. 工具调用 / Agent RL。
3. 金融领域微调。

如果目标是竞赛或公开 benchmark，优先选：

1. LMSYS Chatbot Arena 偏好建模。
2. AIMO / 数学推理。
3. 摘要 RLHF 复现。
