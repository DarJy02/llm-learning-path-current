当前策略已经调整：暂时不继续做更多对比消融实验，先把项目主线完整跑通。

最新主线：

```text
Qwen/Qwen3-1.7B
-> qwen3_nothink
-> 电商客服 SFT
-> DPO 偏好对齐
-> Chat 测试
-> badcase 记录
-> 简历/面试包装
```

重要判断：

1. 之前 `Qwen2.5-1.5B Base` 和 `Qwen3-1.7B-Base` 在少量客服 SFT 下都容易复读，原因不是单纯训练时间不够，而是 Base 小模型缺少稳定聊天边界，加上客服模板句高频，容易学成套话循环。
2. 当前不再硬搞 Base 版本，主线使用 `Qwen/Qwen3-1.7B`，也就是 Qwen3 的 post-trained checkpoint。注意不要用 `Qwen/Qwen3-1.7B-Base`。
3. `Qwen3-1.7B` 虽然不叫 Instruct，但不是纯 Base；在 LLaMA-Factory 里对应 `Qwen3-1.7B-Thinking`，推理和训练 template 用 `qwen3_nothink`。

### DPO 后检查重点

DPO 跑完后不要只看 loss，要直接 Chat 测试：

1. 是否还会一直吐 token 到 max。
2. 是否还会重复 `亲亲/您好/还有其他可以帮您吗`。
3. 是否能先回答客户问题，而不是堆套话。
4. 是否能给出明确操作路径、材料、条件和时效。
5. 是否过度变冷淡，完全不像客服。

建议推理参数：

```text
temperature: 0.7
top_p: 0.8
repetition_penalty: 1.12
presence_penalty: 1.5
max_new_tokens: 128
```

### 面试讲法

当前项目可以这样讲：

> 初始尝试用 Base 小模型做电商客服 SFT，但在低资源客服数据下出现了重复输出和停止不稳定问题。分析后发现，问题主要来自 Base 模型聊天边界弱、客服模板句占比高、数据中存在寒暄和低信息轮次。因此后续将主线切换到 Qwen3-1.7B post-trained checkpoint，并重新清洗数据为 v7_quality，只保留能解决客户问题的样本。SFT 后进一步构造 300 条偏好数据，chosen 使用可执行客服回答，rejected 使用模板化或无效回答，通过 DPO 将模型从“像客服”进一步对齐到“能解决客户问题”。
