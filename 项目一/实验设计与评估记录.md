## 1. 当前主线与 Baseline

当前主线已经从旧的 `Qwen2.5-1.5B-Instruct + q_proj,v_proj` 切换为：

```text
Qwen2.5-1.5B（Base）
+ LoRA

+ 数据集 = ecommerce_customer_sft
```

## 2. 已完成的 Rank 消融实验

本轮实验固定：

```text
模型 = Qwen2.5-1.5B
target_modules = all
dataset = ecommerce_customer_sft
samples = 5773
epoch = 2
cutoff_len = 512
batch_size = 1
gradient_accumulation = 8
learning_rate = 1e-4
lora_dropout = 0.05
val_size = 0.05
```

只改变 LoRA rank 和对应 alpha。

| 实验名 | rank | alpha | eval_loss | ppl | train_loss | adapter 大小 | 训练目录 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| baseline_r4_all | 4 | 16 | 1.7822 | 5.9432 | 2.1194 | 17.66 MB | `train_2026-04-25-00-31-Qwen-1.5B-sft-r4-all` |
| ablation_r8_all | 8 | 16 | 1.7747 | 5.8983 | 2.1188 | 35.27 MB | `train_2026-04-23-14-19-Qwen2.5-1.5B-sft-r8-all` |
| ablation_r16_all | 16 | 32 | 1.6166 | 5.0358 | 1.9840 | 70.49 MB | `train_2026-04-25-15-22-Qwen2.5-1.5B-sft-r16-all` |
| ablation_r32_all | 32 | 64 | 1.3833 | 3.9880 | 1.8158 | 140.92 MB | `train_2026-04-25-15-22-Qwen2.5-1.5B-sft-r32-all` |

### 2.1 当前实验观察

1. `r=4` 和 `r=8` 差距很小。  
   这说明在当前客服数据上，低 rank 已能学到大量基础客服模式。

2. `r=16` 和 `r=32` 的 loss 明显更低。  
   这说明在 `target=all` 的设定下，更大 rank 的确增强了拟合能力。

3. 代价也很明显。  
   adapter 体积从 `17.66 MB -> 140.92 MB`，已经扩大了 8 倍。

4. 当前 baseline 仍保留 `r=4`。  
   原因是项目现阶段优先做“低成本可复现基线 + 消融对比”，不直接把最重模型当标准答案。

### 2.2 当前结论写法

可以这样写：

> 在 `Qwen2.5-1.5B` 基座模型上，以 `target_modules=all` 为设定完成 LoRA rank 消融。结果显示，`r=4` 与 `r=8` 在验证集 loss 上差异很小，说明低 rank 已能覆盖主要客服话术模式；进一步提升到 `r=16` 和 `r=32` 后，eval_loss 明显下降，但 adapter 体积和训练成本显著上升。因此本地 baseline 采用 `r=4` 保持轻量可复现，同时保留 `r=16/r=32` 作为更强效果对照组。

## 3. 模型评估设计

项目一评估不只看 loss，而是分成四类：

```text
1. eval_loss / ppl
2. 自动任务指标（BLEU / ROUGE）
3. 人工评测
4. A/B 对比
```

### 3.1 Eval Loss / PPL

记录：

```text
train_loss
eval_loss
ppl = exp(eval_loss)
loss 曲线走势
是否出现平台化或反弹
```

用途：判断训练是否学到了验证集分布，是否过拟合。

### 3.2 自动任务指标

建议在固定评测集上，对 Base / r4 / r8 / r16 / r32 统一生成答案，然后计算：

```text
BLEU-4
ROUGE-1
ROUGE-2
ROUGE-L
```

注意：客服回答存在多种正确说法，BLEU/ROUGE 只作为辅助，不作为唯一结论。

### 3.3 人工评测

建议按 1-5 分打分，维度如下：

| 维度 | 说明 |
| --- | --- |
| 准确性 | 是否真正回答了用户问题 |
| 可执行性 | 是否给出明确操作步骤、条件、时效 |
| 客服语气 | 是否礼貌、安抚、像真实客服 |
| 上下文一致性 | 是否正确利用历史对话 |
| 幻觉风险 | 是否编造政策、赔付或时效 |

### 3.4 A/B 对比

当前推荐 A/B 组合：

```text
Qwen2.5-1.5B Base vs baseline_r4_all
baseline_r4_all vs ablation_r8_all
baseline_r4_all vs ablation_r16_all
baseline_r4_all vs ablation_r32_all
```

记录：

```text
哪边胜
是否打平
是否都差
典型 badcase
```

## 5. 当前推荐执行顺序

1. 用固定 50 条评测集，先跑 `Qwen2.5-1.5B Base` 和 `baseline_r4_all`。
2. 再统一跑 `r8 / r16 / r32` 的批量推理输出。
3. 先看自动指标，再做人工 A/B。
4. 如果 `r16/r32` 的自动指标明显更好，再决定是否把主 baseline 从 `r4` 升级到 `r16`。
5. 如果人工评测发现 `r4` 和 `r16/r32` 差距并不大，就继续保留 `r4` 作为主 baseline。

## 6. Chat 推理参数与已发现问题

### 6.1 推荐 Chat 推理参数

为了保证 Base / r4 / r8 / r16 / r32 之间公平对比，Chat 或批量推理时统一使用低随机性的客服参数：

```text
Maximum new tokens: 160
Temperature: 0.2
Top-p: 0.8
Repetition penalty: 1.15
Enable thinking: 关闭
Skip special tokens: 开启
Escape HTML tags: 开启
Chat template: qwen
```

如果仍出现重复输出，则进一步收紧：

```text
Maximum new tokens: 128
Temperature: 0.1
Top-p: 0.7
Repetition penalty: 1.2
```

### 6.2 已发现问题：重复输出直到 token 上限

现象：

```text
模型在 Chat 推理中反复输出同一段客服话术，直到达到 max_new_tokens 才停止。
```

主要原因：

```text
1. 当前主线使用 Qwen2.5-1.5B Base，聊天停止行为弱于 Instruct 模型。
2. max_new_tokens 偏大时，模型没有主动结束就会生成到上限。
3. temperature / top_p 偏高时更容易发散。
4. repetition_penalty 不足时，重复片段不会被有效惩罚。
5. 客服训练数据中存在高频模板句，模型可能学到模板循环。
```

处理方式：

```text
1. 统一降低 temperature 和 top_p。
2. 增加 repetition_penalty。
3. 将 max_new_tokens 从 256 降到 128-160。
4. 关闭 Enable thinking。
5. 正式评测中把重复输出记录为 badcase。
```

详细记录见：

```text
项目一/评测结果/badcase_log.md
```

## 7. ecommerce_customer_sft_v2 数据修正

### 7.1 问题定位

`r=4 / r=8 / r=16 / r=32` 均出现重复句式，说明问题不是单个 rank 设置导致，而更可能来自共同的数据分布和 Base 模型停止行为。

对 v1 数据统计后发现：

```text
原始样本数：5773
unique output：4684
精确重复 output 比例：约 18.86%
高频模板：
- “有什么问题我可以帮您处理或解决呢?” 出现 172 次
- “请问还有其他还可以帮到您的吗?” 出现 131 次
- “感谢您对京东的支持，祝您生活愉快，再见!” 出现 74 次
```

这说明 v1 更像“客服逐轮聊天流水”，包含较多寒暄、收尾、等待查询、短确认和纯订单占位符样本，不完全等价于“业务问题 -> 可执行客服答复”。

### 7.2 v2 处理策略

新增数据构建脚本：

```text
scripts/build_ecommerce_sft_v2.py
```

生成并注册数据集：

```text
LLaMA-Factory/data/ecommerce_customer_sft_v2.json
dataset name: ecommerce_customer_sft_v2
```

主要规则：

```text
1. 删除纯寒暄、纯收尾、评价/打赏、等待查询、短确认类 output。
2. 删除低信息 input，例如“你好”“谢谢”“好的”“没有了”和纯订单占位符。
3. 保留包含业务关键词与可执行动作的样本。
4. 对完全相同 output 做限频，单句最多保留 5 条。
5. 少量补充 curated SOP 样本，但不把固定评测集原样加入训练，避免评测泄漏。
```

当前 v2 结果：

```text
v1 样本数：5773
v2 样本数：2529
其中 JDDC 过滤保留：1161
其中 curated / generated SOP 补充：1368
精确重复 output 比例：约 0.79%
```

详细处理报告：

```text
项目一/数据处理/ecommerce_customer_sft_v2_report.md
```

### 7.3 后续训练建议

`ecommerce_customer_sft_v2` 作为当前主线的数据修正版，用于确认重复输出是否明显缓解，并补足更多客服业务场景。

推荐下一步：

```text
Qwen2.5-1.5B Base + LoRA r=4 + ecommerce_customer_sft_v2
epoch = 2 或 3
其他参数先沿用 baseline_r4_all
```

如果 v2 明显减少复读，再视结果扩展到 3000-5000 条更高质量业务解决型样本；如果 v2 仍然复读严重，则增加一组 `Qwen2.5-1.5B-Instruct + LoRA` 对照，验证 Base 模型停止行为的影响。
