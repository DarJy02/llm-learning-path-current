# 本地训练规模建议

## 当前已下载数据

本地文件：

```text
项目三/data/openr1-math-220k/default/default-00000-of-00010.parquet
```

文件大小：

```text
约 214 MB
```

统计结果：

| 项目 | 数值 |
|---|---:|
| 行数/题目数 | 9,374 |
| 每题 generation 数量 | 平均 2.07 |
| 每题正确 generation 数量 | 平均 1.33 |
| 有正确 + 错误 generation 的题 | 1,160 |
| 有 2 条及以上正确 generation 的题 | 5,374 |
| 没有正确 generation 的题 | 2,840 |
| 可构造 DPO pair 的题，粗略估计 | 6,534 |
| 题目长度 p50/p90/p95 | 235 / 472 / 584 字符 |
| generation 长度 p50/p90/p95 | 13,007 / 32,792 / 41,315 字符 |

结论：

- 一个分片已经够本地第一轮实验。
- generation 很长，`cutoff_len=4096` 会截断一部分样本。
- 不建议一上来训练全部 9,374 条。
- 第一版重点是验证 SFT/DPO 是否改善格式、长度、可解析率和小幅正确率。

## 建议训练规模

### 第一轮：快速验证

用途：

- 确认数据转换没问题。
- 确认 LLaMA-Factory 能跑。
- 看 loss 是否正常下降。

建议：

| 阶段 | 样本数 | cutoff_len | epoch |
|---|---:|---:|---:|
| SFT | 200 - 500 | 2048 | 1 |
| DPO | 200 - 500 pair | 2048 | 1 |

这轮不看最终效果，只看流程。

### 第二轮：本地正式小实验

用途：

- 得到 Base / SFT / SFT+DPO 的第一组可比结果。
- 观察 boxed 格式、输出长度、无法解析率。

建议：

| 阶段 | 样本数 | cutoff_len | epoch |
|---|---:|---:|---:|
| SFT | 1,000 - 2,000 | 4096 | 1 - 2 |
| DPO | 1,000 pair | 4096 | 1 |

这是当前最推荐的配置。

### 第三轮：本地扩大实验

用途：

- 如果第二轮效果正常，再验证数据规模扩大是否有收益。

建议：

| 阶段 | 样本数 | cutoff_len | epoch |
|---|---:|---:|---:|
| SFT | 3,000 - 5,000 | 4096 | 1 |
| DPO | 2,000 - 3,000 pair | 4096 | 1 |

这轮已经会比较慢。只有在前两轮结果值得继续时再做。

## 为什么不建议先训全部 9,374 条

原因：

- generation 很长，p50 已经约 13k 字符。
- 本地训练时很多样本会被 `cutoff_len=4096` 截断。
- 全部训练会明显增加时间，但不一定带来成比例收益。
- 你现在还没有完整 baseline/评估脚本，先大训容易不知道效果来自哪里。

更稳的路线是：

```text
200/500 smoke run
-> 1k/2k 本地正式小实验
-> 3k/5k 扩大实验
-> 服务器 7B 正式实验
```

## 关于 cutoff_len

当前建议：

```text
smoke run: 2048
正式本地: 4096
服务器: 8192
```

如果本地显存只有 8GB - 12GB：

```text
cutoff_len = 2048
```

如果本地显存 16GB - 24GB：

```text
cutoff_len = 4096
```

如果服务器 40GB/80GB：

```text
cutoff_len = 8192
```

## 关于 think 标签

这个分片里的 `generations` 已经包含：

```text
<think>
...
</think>
```

所以第一版可以保留 `<think>`。这意味着 SFT 会让 Qwen2.5-Math 学习 R1 风格的显式推理输出。

注意：

- `template: qwen` 只是 Qwen2.5 的对话模板。
- `<think>` 是训练输出文本的一部分。
- 不需要使用 `qwen3_nothink`。

## 当前推荐

先跑：

```text
SFT: 1000 条
DPO: 1000 pair
cutoff_len: 4096
SFT epoch: 2
DPO epoch: 1
```

如果显存吃紧，退一步：

```text
SFT: 500 条
DPO: 500 pair
cutoff_len: 2048
SFT epoch: 1
DPO epoch: 1
```

第一版结果重点看：

- Base -> SFT：格式和正确率变化。
- SFT -> DPO：输出长度和可解析率变化。
- DPO 后正确率是否没有明显下降。
