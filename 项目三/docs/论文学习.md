# 论文学习

本文件只保留和项目三强相关的核心论文/高影响技术报告/一线开源方案，不放低质量期刊和泛泛综述。

项目三主线：

```text
Qwen/Qwen-Math 小模型
-> 数学推理 SFT
-> DPO 控制长度与格式
-> GRPO/RLVR 可验证奖励
-> TIR/Python 工具辅助推理
-> MATH-500/AIME/AIMO 评估
```

## 最建议的阅读顺序

### 第一周：服务当前 SFT + DPO

1. CoT
2. Self-Consistency
3. MATH
4. Qwen2.5-Math
5. LoRA
6. QLoRA
7. DPO
8. AIMO2 第二名方案
9. OpenR1-Math-220k 数据卡

读完这一组，应该能解释：

- 为什么 SFT 会让输出变长。
- 为什么要看 `\boxed{}` 成功率和可解析率。
- 为什么 DPO 可以用来压缩冗长推理。
- 为什么 chosen/rejected 不能乱构造。

### 第二周：服务 GRPO/RLVR

1. PPO
2. InstructGPT
3. DeepSeekMath
4. DeepSeek-R1
5. Kimi k1.5
6. Open-R1
7. Let's Verify Step by Step
8. Math-Shepherd

读完这一组，应该能解释：

- GRPO 和 PPO 的关系。
- RLVR 为什么适合数学。
- rule-based reward 的优势和 reward hacking 风险。
- 为什么 R1-Zero 能诱发推理，但可读性会出问题。

### 第三周：服务 TIR/代码辅助推理

1. PAL
2. Program of Thoughts
3. ToRA
4. ReAct
5. Toolformer
6. NuminaMath-TIR

读完这一组，应该能解释：

- 哪些数学题适合 Python 辅助。
- TIR 和普通 CoT SFT 的数据格式差异。
- 代码执行结果怎么进入最终答案。
- 为什么 TIR 对数值计算题尤其有用。

## 当前最核心的实验问题

这些问题可以直接变成项目三报告里的研究问题。

1. SFT 提升的是数学正确率，还是主要提升输出格式？
2. OpenR1-Math-220k 的长 reasoning trace 对 1.5B 小模型是否过长？
3. `cutoff_len=4096` 是否会截断最终答案，导致模型学不到 `\boxed{}`？
4. DPO 是否能在不明显降低正确率的前提下降低输出长度？
5. DPO pair 用“正确 vs 错误”更好，还是“正确短 vs 正确长”更好？
6. `<think>` 对 Qwen2.5-Math 是真实推理增益，还是主要是输出风格迁移？
7. 多采样 self-consistency 能否显著弥补小模型 pass@1 不足？
8. rule-based reward 是否会导致模型只学会格式而不真的算对？
9. TIR 是否比纯 CoT 更适合数值计算题？

## 别读的类型

暂时不要花时间在这些上面：

- 没有代码、没有数据、没有强基准的普通期刊论文。
- 只是在现有方法上换名字的 alignment 小变体。
- 泛泛的 LLM survey。
- 没有和数学推理、SFT、DPO、RLVR、TIR 直接关系的 agent 论文。
- 标题很大但实验只在小模型/小数据上做玩具验证的文章。

当前项目最值钱的是：

```text
核心论文读懂
-> 实验跑通
-> 指标对齐
-> 用论文解释自己的实验现象
```
