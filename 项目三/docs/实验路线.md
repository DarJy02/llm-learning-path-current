# 实验路线

## 第一版：SFT + DPO

目标：

> 用 Qwen 小模型完成数学推理 SFT + DPO，并验证训练是否提升数学正确率、答案格式和输出效率。

### Step 1：准备 baseline

模型候选：

- 本地：`Qwen/Qwen2.5-Math-1.5B`
- 服务器：`Qwen/Qwen2.5-Math-7B`

评估集：

- `MATH-500`
- AIMO2 里的 reference/AIME 小集合

记录指标：

- 正确率。
- `\boxed{}` 格式成功率。
- 平均输出长度。
- 超长输出比例。
- 答案无法解析比例。

### Step 2：SFT

数据：

- `OpenR1-Math-220k/default`

训练目标：

- 学会数学长推理。
- 学会最终答案格式。
- 提升 pass@1。

预期风险：

- 输出明显变长。
- 小模型可能学到形式但没有真正提升正确率。
- 长上下文显存压力较大。

### Step 3：DPO

数据构造：

- 同一道题里选择正确且长度适中的 trace 作为 chosen。
- 选择错误、过长、不可解析的 trace 作为 rejected。

训练目标：

- 保持正确率。
- 缩短输出。
- 提高答案可解析率。

参考 AIMO2：

- AIMO2 使用 DPO 主要是为了降低输出长度。
- DPO pair 可以按正确性、最小长度、长度比例、语义相似度筛选。

### Step 4：评估和分析

对比：

| 模型 | 训练阶段 | 评估 |
|---|---|---|
| Base | 未训练 | baseline |
| SFT | SFT 后 | 看正确率和输出长度 |
| SFT + DPO | DPO 后 | 看长度、格式和正确率是否平衡 |

核心问题：

- SFT 是否提升了数学推理能力。
- SFT 是否导致输出变长。
- DPO 是否能压缩输出长度。
- DPO 是否伤害正确率。
- 模型是否稳定输出 `\boxed{}`。

## 第二版：GRPO/RLVR

工具：

- Open-R1。

模型候选：

- `Qwen/Qwen3-1.7B-Base`
- `Qwen/Qwen3-4B-Base`
- `Qwen/Qwen2.5-Math-7B`

Reward 设计：

- 正确答案：主 reward。
- `\boxed{}` 格式：辅助 reward。
- 可解析：辅助 reward。
- 超长惩罚：辅助 reward。

研究问题：

- Rule-based reward 是否能超过 DPO。
- 小模型是否会 reward hacking。
- 答案格式奖励是否会带来虚假规范输出。
- GRPO 是否比 SFT 更依赖采样数量和 prompt。

## 第三版：TIR / 代码辅助推理

数据：

- `NuminaMath-TIR`

目标：

- 让模型学会写 Python 代码辅助求解。
- 比较纯 CoT 和 TIR 在数学题上的差异。

研究问题：

- TIR 是否提升数值型题目正确率。
- 代码执行错误如何影响最终答案。
- 小模型是否能稳定生成可运行代码。

## 当前行动建议
