# 科研项目打造1：MiniMind 小模型长上下文关键信息保持能力研究

## 1. 项目定位

这个项目的目标不是直接复现邱锡鹏老师团队的某个完整项目，也不是做一个老师已有项目的“阉割版”。更合适的思路是：从 OpenMOSS / 邱老师团队多个方向中抽取一个共同的底层问题，然后围绕这个问题做一个独立、可落地、能解释 match 的小型科研项目。

目前建议的切入口是：

> 小模型在长上下文中如何保持关键信息？当上下文变长、干扰信息增加、关键信息位置变化时，模型为什么会失败？能否通过可控数据构造和微调提升这种能力？

这个方向和 OpenMOSS 的以下研究线有联系：

- L-Eval：长上下文模型评测。
- LongSafety：长上下文安全问题。
- AgentLongBench：长上下文智能体环境 rollout。
- MOSS / OpenMOSS：工具增强、智能体、后训练、评测。
- MiniMind：适合作为低资源实验平台的小模型。

因此，项目可以先定位为：

> **MiniMind-LongContext：小模型长上下文关键信息保持能力研究**

后续如果实验顺利，可以升级为：

> **面向长上下文智能体的关键状态记忆机制研究**

## 2. 为什么不直接复现老师项目

直接复现老师项目存在几个问题：

1. 完整项目通常系统很大，个人短期内很难复现到位。
2. 如果只是跑通代码，很难体现自己的研究判断。
3. 如果做一个简化版，容易被看成“阉割版复现”，创新性不强。
4. 老师更可能关注：你是否理解该方向背后的核心问题，并能把问题拆成可研究、可验证的小切片。

更好的叙事是：

> 我关注邱老师团队在长上下文、智能体和评测方面的工作。受到 L-Eval、LongSafety 和 AgentLongBench 的启发，我选择“小模型长上下文关键信息保持”作为切入点，构造可控实验，分析长度、位置、干扰和微调数据对模型表现的影响。

这样既能体现匹配，又不是简单复制已有项目。

## 3. 核心研究问题

主问题：

> 小模型在长上下文任务中丢失关键信息的主要原因是什么？长度、位置、干扰信息、冲突信息分别会带来怎样的影响？

可以拆成四个子问题：

1. 上下文越长，MiniMind 的关键信息提取能力是否稳定下降？
2. 关键信息出现在开头、中间、结尾时，模型是否存在明显位置偏置？
3. 当干扰信息和关键信息语义相似时，模型是否更容易答错？
4. 通过 SFT 微调，能否提升 MiniMind 对长上下文关键事实的保持能力？

可选进一步问题：

5. 让模型先抽取证据再回答，是否比直接回答更稳？
6. 针对“状态更新”类任务，模型能否识别最新信息覆盖旧信息？
7. 微调后模型是否只是记住合成模板，还是能泛化到新模板、新长度、新干扰类型？

## 4. 相关论文学习路线

### 第一组：理解长上下文为什么难

1. **Lost in the Middle: How Language Models Use Long Contexts**
   - 重点：模型在长上下文中更容易忽略中间位置的信息。
   - 学习目标：理解“位置偏置”这个核心现象。

2. **LongBench: A Bilingual, Multitask Benchmark for Long Context Understanding**
   - 重点：长上下文评测有哪些任务类型。
   - 学习目标：理解长文档 QA、多文档 QA、摘要、推理等评测形态。

3. **L-Eval: Instituting Standardized Evaluation for Long Context Language Models**
   - 重点：邱老师团队相关，长上下文标准化评测。
   - 学习目标：理解如何构造更系统的长上下文 benchmark。

### 第二组：理解评测如何变难

4. **RULER: What’s the Real Context Size of Your Long-Context Language Models?**
   - 重点：模型标称支持很长上下文，不代表真实能用那么长。
   - 学习目标：学习可控合成任务设计。

5. **NoLiMa: Long-Context Evaluation Beyond Literal Matching**
   - 重点：长上下文评测不能只做字面匹配，还要看隐式关联。
   - 学习目标：避免项目只停留在“找字符串”的简单层面。

6. **LongSafety: Evaluating Long-Context Safety of Large Language Models**
   - 重点：长上下文下安全问题会更复杂。
   - 学习目标：理解长上下文中的隐藏风险、干扰信息和安全判断。

### 第三组：后续连接智能体

7. **AgentLongBench: A Controllable Long Benchmark For Long-Contexts Agents via Environment Rollouts**
   - 重点：长上下文 agent 中，历史轨迹和工具反馈会不断累积。
   - 学习目标：理解从静态长文档到动态 agent 轨迹的升级。

8. **In-Memory Learning: A Declarative Learning Framework for Large Language Models**
   - 重点：LLM agent 如何从历史经验中形成自然语言 memory。
   - 学习目标：为后续“关键状态记忆”扩展做准备。

## 5. 数据构造设计

第一阶段不需要真实复杂数据，先做可控合成数据。

每条样本包含：

- 一段长上下文；
- 一个或多个关键事实；
- 若干干扰事实；
- 一个问题；
- 标准答案；
- 可选的证据句。

示例：

```text
背景文本：
用户 A73 在第一轮测试中的权限码是 1842。
用户 B19 的权限码是 3901。
用户 C88 的权限码是 7720。
……
经过最终审核，用户 A73 的最终权限码被更新为 4912。
……

问题：
用户 A73 的最终权限码是什么？

答案：
4912
```

可以控制的变量：

- 上下文长度：512 / 1k / 2k / 4k / 8k token。
- 关键事实位置：开头 / 中间 / 结尾。
- 干扰类型：无关干扰 / 相似干扰 / 冲突干扰。
- 任务类型：单事实提取 / 多事实组合 / 状态更新 / 多跳推理。
- 输出形式：直接答案 / 先证据后答案。

## 6. 实验设计

### Baseline 1：未微调 MiniMind

直接测试 MiniMind 在不同长度、不同位置、不同干扰类型下的准确率。

关注现象：

- 长度增加是否导致准确率下降；
- 中间位置是否最容易失败；
- 相似干扰是否比无关干扰更难；
- 状态更新任务是否明显更难。

### Baseline 2：普通 QA 微调

用短上下文 QA 数据微调 MiniMind，然后测试长上下文任务。

目的：

> 验证普通问答能力提升是否能自然迁移到长上下文关键信息保持。

### Baseline 3：长上下文 QA 微调

用合成的长上下文 QA 数据微调 MiniMind。

目的：

> 验证针对性数据构造是否能提升长上下文能力。

### Method：证据抽取式微调

训练模型先输出证据句，再输出答案。

输出格式：

```text
证据：经过最终审核，用户 A73 的最终权限码被更新为 4912。
答案：4912
```

假设：

> 相比直接回答，显式证据抽取可以帮助小模型更稳地定位关键信息，减少被干扰事实误导。

## 7. 评测指标

基础指标：

- Accuracy：最终答案是否正确。
- Exact Match：答案是否完全匹配。
- Position Accuracy：不同位置下的准确率。
- Length Robustness：上下文变长后的性能下降幅度。
- Distractor Robustness：干扰增强后的性能下降幅度。

如果加入证据输出：

- Evidence Recall：输出证据是否包含标准关键句。
- Evidence-Answer Consistency：证据和答案是否一致。
- Conflict Error Rate：是否被旧信息或冲突信息误导。

## 9. 6-8 周执行计划

### 第 1 周：论文入门和项目框架

- 阅读 Lost in the Middle、LongBench、L-Eval。
- 整理长上下文评测的基本任务类型。
- 确定 MiniMind 模型版本和训练环境。

### 第 2 周：合成数据生成

- 写数据生成脚本。
- 生成单事实提取任务。
- 控制长度、位置、干扰类型。
- 人工检查 50 条样本质量。

### 第 3 周：Baseline 测试

- 跑未微调 MiniMind。
- 统计不同长度、位置、干扰下的准确率。
- 初步画图，找出失败模式。

### 第 4 周：普通 QA 微调和长上下文 QA 微调

- 构造训练集。
- 跑 SFT。
- 对比微调前后结果。

### 第 5 周：证据抽取式微调

- 构造“证据 + 答案”格式训练数据。
- 微调 MiniMind。
- 测试 evidence recall 和 answer accuracy。

### 第 6 周：错误分析和补充实验

- 分析模型是否被相似干扰误导。
- 分析状态更新任务上的错误。
- 测试模板外泛化能力。

### 第 7-8 周：整理项目

- 写 README。
- 整理实验图表。
- 写技术报告。
- 准备一页项目介绍，用于联系导师或面试展示。

## 10. 对导师匹配的表达方式

可以这样表述：

> 我关注邱锡鹏老师团队在长上下文评测、智能体和后训练方向的工作。受到 L-Eval、LongSafety 和 AgentLongBench 的启发，我没有直接复现完整系统，而是选择“小模型长上下文关键信息保持”作为切入点。基于 MiniMind，我构造了可控长上下文数据，分析上下文长度、关键信息位置、干扰类型对模型表现的影响，并尝试通过证据抽取式微调提升模型的关键事实定位能力。

这个表达的优势：

- 和老师方向相关；
- 没有直接复制老师项目；
- 项目边界清楚；
- 能体现你已有 MiniMind 和微调基础；
- 后续可以自然扩展到 agent memory、长上下文安全或长上下文评测。

## 11. 当前阶段的判断

这个项目适合作为第一阶段科研训练项目，因为：

1. 与已有知识储备衔接：MiniMind + 微调训练。
2. 算力需求相对可控。
3. 有明确实验变量。
4. 有相关论文可学。
5. 与 OpenMOSS 长上下文、智能体、评测方向存在自然联系。
6. 可以从简单版本开始，逐步扩展，而不是一开始背负一个宏大系统。

下一步最关键的不是马上追求“创新很大”，而是先完成一个可复现、可评测、可解释的小闭环。
