# 科研项目打造1 Pro：长上下文智能体的记忆、证据与自我改进机制

## 0. 文档定位

这个文档不是一个短期项目计划，而是一条可以长期阶段性打造的科研项目主线。

`科研项目打造1.md` 更像第一阶段的地基项目：从长上下文中的关键信息保持能力入手，做可控实验、评测和微调。

本 Pro 版本则是长期路线图：在项目1的基础上，逐步扩展到 RAG、Agent、Memory、Self-improvement 和 OpenMOSS 相关研究方向。

核心思想是：

> 先把“长上下文中如何保留关键信息”做扎实，再把静态长文本问题升级为动态智能体任务中的记忆、证据和自我改进问题。

## 1. 长期主线

最适合长期打造的主线可以概括为：

> **长上下文智能体的记忆、证据与自我改进机制**

这条线不是单纯做 RAG，也不是单纯做 Agent，更不是“自动写论文机器”。它关注的是一个更底层的问题：

> 当模型或智能体面对大量信息、多步任务、工具调用历史和反馈轨迹时，它应该记住什么、遗忘什么、压缩什么、验证什么，以及如何从过去的成功和失败中变强？

它可以连接以下方向：

- **长上下文**：L-Eval、LongBench、RULER、Lost in the Middle。
- **RAG**：外部知识检索、证据定位、引用生成。
- **Agent**：工具调用、多步规划、环境反馈、历史轨迹。
- **Memory**：短期记忆、长期记忆、任务记忆、失败经验。
- **RL / Self-improvement**：RLoop、In-Memory Learning、DPO/RLHF、trajectory learning。
- **OpenMOSS match**：长上下文、工具增强、智能体、后训练、评测。

## 2. 项目1和 Pro 路线的衔接

你完全可以先打造好 `科研项目打造1.md` 里的路线，再逐步接到 Pro 版本。这不是两个割裂方向，而是递进关系。

### 2.1 项目1是什么

项目1关注：

> 模型能不能在长上下文中找到、保持和使用关键信息？

它偏基础实验，主要产出包括：

- 长上下文数据构造能力。
- 位置、长度、干扰强度等变量控制。
- 模型评测脚本。
- SFT 微调经验。
- long-context failure taxonomy。
- 证据抽取式回答实验。

### 2.2 Pro 版本是什么

Pro 版本关注：

> 当上下文不再是一篇静态文档，而是一个智能体的多步任务轨迹时，关键状态和证据如何被保留、压缩、调用和更新？

也就是把：

> 长文档里的关键信息

升级为：

> Agent 历史轨迹里的关键状态、工具证据、失败经验和任务约束。

### 2.3 自然升级路径

两者之间的衔接可以这样理解：

1. **项目1**：模型能不能在长上下文里找到关键事实？
2. **升级1**：如果上下文不是文档，而是 agent 的历史轨迹呢？
3. **升级2**：哪些历史轨迹应该进入 memory？
4. **升级3**：memory 如何区分证据、任务状态、偏好、失败经验？
5. **升级4**：agent 能不能从成功/失败轨迹中自我改进？
6. **升级5**：能不能迁移到某个具体领域的长期任务 agent？

因此，项目1是地基，Pro 是长期上层建筑。

## 3. RAG、Agent、Memory、RL 的关系

这一部分先把概念摆清楚，避免项目变成一团混乱。

### 3.1 RAG 是什么

RAG 是一种增强模型知识和证据能力的技术。

典型流程是：

> 用户问题 → 检索相关文档 → 把文档放进 prompt → 模型基于文档回答

RAG 主要解决：

- 模型不知道最新知识。
- 模型需要引用外部证据。
- 模型参数内知识不可靠。
- 需要从大量文档中定位相关内容。

### 3.2 Agent 是什么

Agent 是更大的概念。它不是只回答一次问题，而是能围绕目标进行多步行动。

典型流程是：

> 目标 → 规划 → 调用工具 → 观察结果 → 更新计划 → 再调用工具 → 最终完成任务

Agent 可以调用很多工具：

- search / paper retrieval。
- pdf reader。
- code executor。
- calculator。
- web browser。
- vector database。
- memory。
- verifier。
- planner。

### 3.3 RAG 和 Agent 的关系

RAG 可以看作 Agent 的一个工具。

如果只是问答：

> 检索文档 → 回答问题

这就是比较普通的 RAG。

如果任务变成：

> 拆解问题 → 多轮检索 → 比较证据 → 调用代码验证 → 反思错误 → 重新检索 → 输出结果

这就进入 Agent 范畴。

### 3.4 Memory 是什么

Memory 解决的是跨步骤、跨任务的信息保留问题。

不是所有历史都应该塞进上下文。一个更合理的智能体需要区分：

- **task memory**：当前任务目标、约束、阶段状态。
- **evidence memory**：关键证据、来源、引用。
- **failure memory**：失败尝试、错误原因。
- **preference memory**：用户偏好、导师方向、项目风格。
- **skill memory**：某类任务的有效做法。

### 3.5 RL / Self-improvement 是什么

Self-improvement 关注：

> 智能体能不能从过去成功或失败的轨迹中变强？

可能有两种路径：

- **参数更新**：把成功轨迹变成训练数据，对模型做 SFT / DPO / RL。
- **记忆更新**：不改模型参数，而是更新外部 memory，让下一次任务表现更好。

一个很重要的长期问题是：

> 对于长程任务，什么时候应该微调模型，什么时候只需要更新 memory？

## 4. 三条长期路线

### 4.1 路线 A：长上下文智能体记忆系统

核心问题：

> Agent 在长程任务中如何选择、压缩、更新和调用记忆？

阶段路径：

1. 学 RAG：BM25、embedding、向量数据库、chunk、rerank。
2. 学长上下文评测：Lost in the Middle、LongBench、L-Eval、RULER。
3. 学 Agent：ReAct、Toolformer、Reflexion、AutoGPT 类框架。
4. 学 Memory：MemGPT、In-Memory Learning、episodic memory、semantic memory。
5. 做自己的系统：一个能执行长期任务、记录历史、调用记忆、验证证据的 agent。

潜在创新点：

> 不只是把所有历史塞进上下文，而是区分 task memory、evidence memory、failure memory、preference memory，并为不同记忆设计不同的更新和淘汰规则。

### 4.2 路线 B：长上下文证据可信评测

核心问题：

> 长上下文模型或 Agent 生成的答案是否真的被上下文证据支持？

阶段路径：

1. 学长上下文 QA。
2. 学 evidence retrieval / citation。
3. 学 hallucination evaluation。
4. 学 LLM-as-judge。
5. 做一个 benchmark：给长文档、多工具轨迹，让模型回答并引用证据，评测引用是否真实支持结论。

潜在创新点：

> 建立“答案-证据-轨迹一致性”的评测，而不是只看最终答案对不对。

### 4.3 路线 C：可验证任务上的 Agent 自我改进

核心问题：

> Agent 能否从自己过去的成功/失败轨迹中学习？

阶段路径：

1. 学 SFT / LoRA / DPO。
2. 学 ReAct / tool-use。
3. 学 verifiable reward：数学、代码、游戏、检索任务。
4. 学 RLoop / self-improvement / process reward。
5. 做一个 agent：执行任务 → 收集轨迹 → 验证成功 → 过滤轨迹 → 微调或更新 memory → 下一轮提升。

潜在创新点：

> 比较“参数更新”和“记忆更新”两种自我改进方式：是微调模型更有效，还是更新外部 memory 更有效？

## 5. 推荐主路线

如果愿意长期投入，最推荐合并路线 A 和路线 C：

> **面向长程任务的记忆增强智能体：从 RAG、长上下文到自我改进**

但阶段上不要一开始就做“科研智能体”或“自动写论文”。更合适的推进方式是：

1. **阶段一：RAG 基础**
   做一个能检索文档、回答问题、引用来源的系统。

2. **阶段二：长上下文与证据**
   研究当文档很多、上下文很长时，模型如何丢失关键信息，如何压缩和保留证据。

3. **阶段三：Agent 化**
   让系统不只是问答，而是能自己规划、调用工具、观察结果、调整下一步。

4. **阶段四：Memory**
   让 agent 记住过去读过的材料、失败过的尝试、用户偏好、任务约束，而不是每次从零开始。

5. **阶段五：Self-improvement**
   让 agent 根据反馈更新 memory 或训练数据，逐步提升长期任务表现。

## 6. 除了科研论文 Agent，还能做什么 Agent

“科研论文 agent”只是一个可能落地方向，而且不建议一开始包装成“自动写论文机器”。这个说法容易显得浅，也不够严肃。

更好的定位是：

> 长上下文任务智能体的记忆与证据机制研究。

然后选择一个具体场景落地。

### 6.1 代码调试 Agent

任务：

> 读代码、运行测试、定位 bug、修改代码、记录失败尝试。

为什么合适：

- 结果可验证，测试通过就是通过。
- 很适合接 self-improvement 和 RLoop 类思想。
- 历史轨迹很长，包含错误日志、代码片段、失败尝试。

匹配点：

- 工具调用。
- 长上下文。
- 轨迹记忆。
- 可验证反馈。
- 自我改进。

### 6.2 长文档证据审计 Agent

任务：

> 读合同、论文、政策、技术文档，回答问题时必须给证据，并检查答案是否被证据支持。

为什么合适：

- 和项目1的长上下文关键信息保持最自然衔接。
- 不需要一开始做复杂多工具 agent。
- 能体现可信生成和证据一致性。

匹配点：

- L-Eval。
- LongSafety。
- RAG。
- 证据定位。
- 长上下文可信评测。

### 6.3 数据构造与评测 Agent

任务：

> 自动生成 long-context benchmark，控制长度、位置、干扰强度，然后用 verifier 检查数据质量。

为什么合适：

- 它不是自动写论文，而是在做评测基础设施。
- OpenMOSS 很多工作本身就是数据、benchmark、评测驱动。
- 可以和项目1直接结合。

匹配点：

- benchmark construction。
- LLM-as-judge。
- verifier。
- controllable evaluation。
- long-context failure analysis。

### 6.4 实验管理 Agent

任务：

> 记录训练配置、日志、loss、评测结果，根据失败结果建议下一轮实验。

为什么合适：

- 更像真实科研工程助手，不是代替研究者写论文。
- 能结合你已有的大模型微调训练经验。
- 后续可以连接 AI Infra 和后训练。

匹配点：

- 实验日志理解。
- 训练过程分析。
- memory。
- self-improvement。
- research workflow automation。

### 6.5 安全审计 Agent

任务：

> 在长对话或长文档里发现隐藏的风险指令、越权行为、冲突信息。

为什么合适：

- 和 LongSafety 贴合。
- 长上下文中的风险信息通常不是显眼出现，而是隐藏在大量上下文里。
- 可以做成评测 + 方法，不必训练大模型。

匹配点：

- 长上下文安全。
- 隐藏风险定位。
- agent safety。
- evidence tracing。

### 6.6 游戏 / 数学 / 代码类可验证 Agent

任务：

> 做数学题、代码题、小游戏规划，靠规则或测试验证对错。

为什么合适：

- reward 明确，最适合接 RL / RLoop。
- 失败和成功轨迹可以自动筛选。
- 能研究参数更新和 memory 更新哪个更有效。

匹配点：

- verifiable reward。
- trajectory learning。
- RLoop。
- self-improvement。
- RL reasoning。

## 7. Agent 方向优先级建议

如果按照“和项目1衔接自然程度 + 和 OpenMOSS 匹配度 + 个人可落地性”排序：

1. **长文档证据审计 Agent**
   - 最自然接项目1。
   - 适合长上下文、证据一致性、RAG。

2. **数据构造与评测 Agent**
   - 很适合做科研训练和 benchmark。
   - 能反过来服务项目1的数据和评测。

3. **代码调试 Agent**
   - 最可验证，工程含金量高。
   - 适合接 self-improvement。

4. **实验管理 Agent**
   - 长期想象力大。
   - 前期定义要小，否则容易空泛。

5. **安全审计 Agent**
   - 和 LongSafety 贴合。
   - 需要认真设计风险样本。

6. **科研选题 / 科研论文 Agent**
   - 可以放后期。
   - 不建议一开始就做，避免被理解成“自动写论文机器”。

## 8. 如何在项目1中预留 Agent 接口

如果先做项目1，建议一开始就把数据结构设计得更像 agent trajectory，而不是普通 QA。

除了保存：

- context。
- question。
- answer。
- evidence。

还可以额外保存：

- `step_id`：信息出现在哪一步。
- `source_type`：信息来自文档、工具返回、用户输入还是历史记忆。
- `state_update`：是否更新了旧状态。
- `distractor_type`：干扰类型。
- `conflict_with`：是否和旧信息冲突。
- `must_remember`：是否属于后续任务必须保留的信息。
- `memory_type`：task / evidence / failure / preference / skill。

这样项目1后面就可以自然升级：

> 长文档 QA 数据 → 多步工具轨迹数据 → Agent memory 评测数据。

## 9. 和邱锡鹏老师 / OpenMOSS 的匹配表达

不要表达成：

> 我复现了老师的某某项目。

也不要表达成：

> 我想做一个自动科研写论文 agent。

更好的表达是：

> 我关注 OpenMOSS 在长上下文、工具增强智能体和自我改进方向的研究。我的长期项目希望围绕“长上下文智能体中的关键证据与记忆机制”展开：第一阶段研究模型在长上下文中如何保持关键信息；第二阶段将静态长文本扩展为多步工具调用轨迹，研究哪些历史状态应该进入 memory；第三阶段进一步探索 agent 如何基于可验证反馈更新 memory 或训练数据，实现自我改进。

这套表达的好处：

- 和 OpenMOSS 方向相关。
- 不直接复制已有项目。
- 有长期规划。
- 第一阶段能落地。
- 后续能自然接 Agent、Memory、RL。

## 10. 知识树总结

可以用一句话记住这条路线：

> RAG 是检索证据；Long Context 是处理大量证据；Agent 是多步使用证据和工具；Memory 是跨任务保留经验；RL / Self-improvement 是从反馈中变强。

五个概念连起来，就是：

> **长上下文智能体的记忆、证据与自我改进机制。**

这条线既可以从项目1的小实验做起，也能逐步扩展成一个长期科研项目。
